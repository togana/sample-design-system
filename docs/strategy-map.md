# デザインシステム 1ページ戦略マップ

---

## 1. Context（今何が壊れているか）

| 課題 | 影響 |
|------|------|
| プロダクト間でUIがバラバラ | ユーザーが製品を横断するたびに迷う。ブランドの信頼が下がる |
| 同じ部品を各チームが別々に作っている | 開発工数の重複。ボタン1つに N チーム × M 時間かかっている |
| デザイン(Figma)とコードの色・余白の定義がズレている | デザイナーが渡したものと実装が毎回違う。修正ラリーが発生 |
| アクセシビリティ対応がチーム任せ | 対応レベルに差があり、法的リスクと利用者の不便が放置されている |

---

## 2. Vision（12ヶ月後の姿）

> すべてのプロダクトが **1つの部品棚(コンポーネントライブラリ)** を使っている。
> Figma で色や余白を変えると、コードにも **自動で反映** される。
> どのプロダクトも **同じ水準のアクセシビリティ** を満たしている。

---

## 3. Strategy（3つの賭け）

### 賭け① 土台を固める（Month 1〜3）

- デザイントークン（色・余白・文字サイズ）を Figma → Panda CSS で **一元管理** する仕組みを作る
- Figma Variables → コード自動同期のパイプラインを構築
- Button, Input, Dialog など **最も使われる5部品** を Ark UI ベースで実装し、1つのプロダクトで試す

### 賭け② 実績を作る（Month 4〜8）

- パイロットプロダクト(1〜2チーム)に導入し、**実戦で磨く**
- 部品数を 15〜20 に拡大（Table, Tabs, Toast, Select など）
- 導入チームからのフィードバックを2週間サイクルで反映
- ドキュメントサイト（Next.js）を公開し、使い方・ガイドラインを整備

### 賭け③ 全社展開する（Month 9〜12）

- 残りのプロダクトチームへ順次展開
- 既存の独自実装からの **移行ガイドとサポート体制** を用意
- トークン変更 → Figma/コード同時反映の運用を安定化
- 貢献ガイドラインを整備し、各チームが部品を追加できる状態にする

---

## 4. Metrics（月次で計測する指標）

| 指標 | 計測方法 | 目標（12ヶ月後） |
|------|----------|------------------|
| **採用率** | デザインシステム部品を使っているプロダクトの割合 | 80%以上 |
| **部品カバー率** | 各プロダクトの画面内でDS部品が占める割合 | 60%以上 |
| **トークン同期率** | Figma とコードでトークンが一致している割合 | 100% |
| **独自実装の新規発生数** | DS にある部品を別途作ってしまった件数 | 月0件 |
| **導入チーム満足度** | 四半期アンケート（5段階） | 4.0以上 |
| **アクセシビリティ適合率** | WCAG 2.1 AA 基準の自動テスト通過率 | 95%以上 |

---

## 5. Stakeholders（誰にどう報告するか）

| 相手 | 関心事 | 報告の形 | 頻度 |
|------|--------|----------|------|
| **経営層 / CTO** | 投資対効果、開発速度の向上 | 採用率・重複削減の数字を1枚サマリーで | 月1回 |
| **プロダクトマネージャー** | 自チームの導入スケジュールと影響 | 導入ロードマップ + 移行工数の見積もり | 導入前に1回 + 月次 |
| **デザイナー** | Figma ライブラリの使い方、トークンの更新 | Figma ライブラリの変更ログ + ガイド更新通知 | 変更のつど |
| **エンジニア（各チーム）** | 導入手順、破壊的変更の有無 | リリースノート + マイグレーションガイド | リリースごと |
| **DS チーム内** | 進捗・課題・次のスプリント目標 | 週次スタンドアップ + バックログ確認 | 週1回 |
