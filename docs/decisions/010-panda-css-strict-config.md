# ADR-010: Panda CSS の厳格な設定を導入する

- ステータス: 承認済み
- 日付: 2026-03-07

## コンテキスト

デザインシステムの品質と一貫性を保つため、Panda CSS の設定を見直す必要がある。

現状の課題:

- トークンとして定義していない任意の値（`red`, `123px` 等）がスタイルに使用できてしまう
- `bg`, `pos` 等のショートハンドプロパティが混在し、コードの一貫性が損なわれる可能性がある
- CSS プロパティに無効な値（`display: "abc"` 等）を指定しても型エラーにならない
- 開発環境でもクラス名がハッシュ化・CSS が圧縮されると、デバッグが困難になる
- config のバリデーション失敗が warning のみで見逃される恐れがある

## 決定

以下の設定を `panda.config.ts` に追加する。

### 環境別の最適化

| 設定 | 開発 | 本番 | 目的 |
| --- | --- | --- | --- |
| `minify` | `false` | `true` | 開発時は可読性を優先、本番ではファイルサイズを削減 |
| `hash` | `false` | `true` | 開発時はクラス名を読みやすく、本番では短縮 |

`process.env.NODE_ENV === "production"` で切り替える。

### 厳格性の強化

| 設定 | 値 | 目的 |
| --- | --- | --- |
| `shorthands` | `false` | `bg` → `background` 等の省略記法を禁止し、プロパティ名の一貫性を保つ |
| `strictTokens` | `true` | 定義済みトークン以外の値を型エラーにする |
| `strictPropertyValues` | `true` | CSS プロパティに対して有効な値のみを許可する |
| `validation` | `"error"` | config のバリデーション失敗時に warning ではなく error を投げる |

`strictTokens` と `strictPropertyValues` を併用することで、トークン値と CSS プロパティ値の両面で型安全性を確保する。

### エスケープハッチ

`strictTokens` / `strictPropertyValues` が有効な状態でも、`[値]` 構文で任意の値を指定できる。例外的なケースに限り使用する。

```tsx
// ❌ エラー
css({ fontSize: "123px" })

// ✅ エスケープハッチ
css({ fontSize: "[123px]" })
```

## 根拠

1. **デザインシステムの一貫性**: トークン外の値やショートハンドの混在を型レベルで防ぐことで、コードレビューに頼らずルールを強制できる
2. **開発体験の維持**: `minify` と `hash` を環境別に切り替えることで、開発時のデバッグ効率を損なわない
3. **早期エラー検出**: `validation: "error"` により、config の設定ミスをビルド時に即座に検出できる
4. **柔軟性の確保**: `[値]` によるエスケープハッチがあるため、例外的なケースにも対応できる

## 今後の影響

- 既存コードで未定義トークンやショートハンドを使用している箇所はコンパイルエラーになるため、移行が必要
- 新規コード作成時にトークン定義漏れがあれば、先にトークンを追加する運用フローになる
